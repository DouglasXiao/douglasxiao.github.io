# 深入解析 agent‑loop：打造可扩展 AI 代理系统的实践

## 引言：什么是 agent loop？

在大型语言模型（LLM）逐渐从单纯聊天转向执行复杂任务的过程中，一个核心概念正在浮出水面——**Agent Loop**。官方文档将其描述为一种迭代的控制循环：每一轮代理运行时，它会先调用模型，检查模型回复，如果模型请求调用工具，则执行这些工具并继续循环；如果模型将任务交给其他专长代理，则切换代理；如果模型给出最终答复则结束。这个过程为模型增加了行动能力，使其不再只能生成文本，而可以读取文件、访问网络、运行命令等。这样的架构正是 `agent‑loop` 项目实践的核心。

最近我实现了一个agent loop并且开源了，感兴趣的可以移步[这里](https://github.com/DouglasXiao/agent-loop)。

本文从整体架构和关键技术出发，对 `agent‑loop` 的实现进行系统性梳理。项目起初只是一个简单的流式代理循环，随着需求的发展，逐渐演化出记忆管理、子代理调度、权限控制、并发执行、技能按需加载、任务管理、工作树隔离和团队协作等模块。理解这些演化背后的设计思想，有助于我们构建自己的高可扩展 AI 代理系统。

## Agent Loop 的架构与核心设计

### 控制循环与流式输出

`agent‑loop` 将 OpenAI/Claude 聊天接口包装成一个可重入的循环。每一轮首先构造系统消息（包括人格、工具说明、项目上下文等）和用户消息，然后调用模型的 chat API。模型回复以流式 SSE 方式返回，循环会实时解析增量片段并通过 `emit_sse` 将内容转发给前端，这保证了用户界面的即时反馈。若模型回复包含工具调用，循环会解析 JSON 参数并调用本地工具，工具结果再以 SSE 事件发送，使模型能够继续推理；当模型不再调用工具时，循环结束并返回最终答复。初始版本中的 `_stream_one_completion` 和 `core_agent_loop_streaming` 正是这一逻辑的具体实现。

这种 **流式代理循环** 有两点优势：

1. **及时性**：与等到生成完整答复后再返回不同，SSE 让用户看到模型思考的过程，提升交互体验；
2. **工具可复用**：代理只需要提供工具列表和执行接口，模型通过函数调用方式主动请求，代理循环负责调度。

### 工具系统与权限管理

为让模型能够与外部世界交互，`agent‑loop` 构建了一套工具系统。工具的定义统一包含名称、参数 schema、调用描述和风险级别，由 `tools_registry.py` 整理。基础工具包括：

- **文件操作**：`read_file` 支持偏移/限制和行号，保证一次读取的内容不会过大；`write_file` 会覆盖文件并返回写入摘要；`edit_file` 支持精准替换；`glob_files` 和 `grep_files` 方便在仓库中查找文件和文本；
- **外部访问**：`web_fetch` 在指定字符数限制内抓取网页；
- **系统操作**：`run_terminal_cmd` 在受控情况下运行 shell 命令（需设置 `AGENT_ALLOW_BASH=1`）；
- **网络数据**：`get_weather` 通过天气 API 获取实况；
- **代理调度**：`run_sub_agent` 和 `run_sub_agents_parallel` 用于调用子代理。

每个工具被标记为 `read`、`mutate`、`network`、`system`、`delegate` 等风险类别，代理读取环境变量 `AGENT_TOOL_MODE` 和 `AGENT_ALLOW_BASH` 决定哪些类别可用。`execute_tool` 在真正调用工具前会检查权限，若不符合则返回权限错误，模型可通过 `AGENT_TOOL_MODE` 指示哪些操作被禁用。这种 **权限控制机制** 在确保安全的同时给用户灵活配置空间。

### 多轮对话与记忆管理

当代理在与用户长期互动时，历史消息会迅速耗尽上下文窗口。`agent‑loop` 引入了三层记忆管理策略：

1. **工具结果溢出（spill）**：当工具返回内容过长时，`budget_tool_result_for_messages` 会将完整结果写入 `.claude/memory/spill/` 目录，同时返回一段预览文字和文件路径供模型后续按需读取。这样可以避免一次性把大文件塞入对话。

2. **微缩（micro‑compact）**：在每轮调用模型之前，对旧的 `role=tool` 消息执行轻量压缩，只保留最近 `AGENT_KEEP_RECENT_TOOL_RESULTS` 条，旧消息用一行占位符替代，包含工具名和被省略的字符数。这一操作无需调用模型，几乎没有成本。

3. **摘要压缩与紧急回收**：当估计 token 使用量接近阈值时，`maybe_compress_conversation` 调用模型生成早期会话的摘要，并用一条简短的摘要消息替换那些老消息，摘要中会附带完整 transcript 的磁盘路径。如果 API 拒绝接受 `max_tokens` 参数或回复 `context_length_exceeded` 错误，则执行 `emergency_compact_inplace` 紧急删除最早的非系统消息以腾出空间 [detail](https://github.com/DouglasXiao/agent-loop/commit/981160e.patch#:~:text=and%20caches%20the%20winner%20per,then%20retries%20the%20call%20once)。

此外，`ensure_memory_layout` 在初始化时创建 `.claude/memory` 目录并准备种子文件，如 `MEMORY.md`、`project_structure.md` 和 `user_preferences.md`，供模型在多轮对话中检索。这一 **记忆体系** 使代理在长时间运行中保持高效且不会丢失重要信息。

### 子代理与并行执行

为了处理复杂任务或解耦子流程，`agent‑loop` 实现了 **子代理机制**。当模型调用 `run_sub_agent` 时，主代理会启动一个带独立系统提示和上下文的子进程，通过 `sub_agent.py` 执行任务。子代理返回 `SubAgentResult`，包含结果文本、所用回合数、耗时、使用工具等信息，并对错误进行了详细分类（配置错误、接口错误、超时、达到最大回合、权限拒绝等） [错误分类详情](https://github.com/DouglasXiao/agent-loop/commit/47c76f0.patch#:~:text=%40%40%20,)。此外，`run_sub_agents_parallel` 允许一次并行运行多个子代理任务并汇总结果 [跳转查看](https://github.com/DouglasXiao/agent-loop/commit/47c76f0.patch#:~:text=%2B%20%20%20%20,to_dict)。

除了代理间并行，项目还在主循环中实现了对只读工具的并发调用：当模型一次返回多个 `read` 或 `network` 类工具调用时，`_run_all_tool_calls` 会将它们分组并使用 `ThreadPoolExecutor` 并行处理。这种 **并发调度** 减少了等待时间，提高了代理响应的流畅度。

### 技能按需加载与任务工作流

随着功能增多，将所有操作流程写入系统提示会导致提示膨胀。为此，`agent‑loop` 引入 **技能 (skills)** 概念：开发者可在 `.claude/skills/<name>/SKILL.md` 中编写技能说明文（例如 `git-commit` 或 `code-review`），工具 `list_skills` 提供技能索引，工具 `load_skill` 可按需加载技能正文，这样模型只有在需要时才去读取技能说明。技能正文会包装在 `<skill name="...">…</skill>` 标签中，方便模型区分不同技能内容。

在更高层级上，系统提供了两套工作流管理工具：

- **持久化待办事项**：`todo_write` 存储任务清单并支持设置、添加、更新、完成和清空操作，持久化于 `.claude/todos/current.json`。系统会在若干轮未更新 TODO 时自动提醒模型使用该工具。

- **任务图 (task graph)**：相比简单的 TODO，`task_graph.py` 定义了支持依赖关系的任务节点，每个任务存储在 `.claude/tasks/task_<id>.json`，包含描述、状态、阻塞关系和负责人，工具 `task_create`、`task_update`、`task_complete` 等允许模型创建和维护任务，并在系统提示中渲染任务概要。此机制让模型能管理复杂项目，清晰表达顺序和依赖。

与任务管理配套的是 **后台任务 (background tasks)**：工具 `bg_run` 让模型在后台执行长时间的 shell 命令，立即返回任务 ID；`bg_check` 查询任务状态和输出；循环会自动将完成的后台任务结果注入会话，模型无需主动轮询。通过后台执行，长耗时命令不会阻塞当前回合。

### 工作树和团队协作

`agent‑loop` 还提供了一些用于团队协作的扩展：

- **Git worktree**：在 `.worktrees/<name>` 目录下创建隔离的 Git 工作树，使用分支 `wt/<name>`。工具 `worktree` 支持创建、删除、保留、列出、运行命令和查看事件。如果在创建工作树时绑定了任务，则任务状态会自动变为 `in_progress`，并在删除工作树时可选择将任务标记为完成。此机制允许多个子任务分别在独立环境中迭代开发，避免互相污染。

- **团队邮箱**：`team_mailbox.py` 为每个团队成员创建独立的收件箱，提供注册、发送、广播、读取、预览和列表操作 [查看这里](https://github.com/DouglasXiao/agent-loop/commit/b3634aa.patch#:~:text=team_mailbox.py%20,already%20carries%20%27type%27%20and%20%27from)。通过这一简单的消息传递系统，模型可以跨会话向不同用户发送信息或广播通知。

这些模块展示了代理如何不仅与代码交互，还能在团队层面协调多人的协作和沟通。

### 鲁棒性与可扩展性

为了适应不同提供商和不断变化的 API，`agent‑loop` 在客户端层面提供了灵活的配置：当环境变量 `OPENROUTER_API_KEY` 存在时，代理会自动切换为 OpenRouter 提供商；否则默认使用 OpenAI。提供商的基地址、模型名称、引用信息等都通过环境变量控制。同时引入了 `model_routed` SSE 事件，记录实际路由到的模型及 token 使用量，使用户能监控成本。

在错误处理方面，代理对 API 的不稳定性做了防御：当调用模型时，如果出现 4xx/5xx 错误，会通过 SSE 通知前端而不是抛出异常；当 API 拒绝 `max_tokens` 参数时，会自动回落到 `max_completion_tokens`，并缓存偏好；在极端情况下通过 `emergency_compact_inplace` 删除旧消息保证运行。默认的上下文长度限制、压缩比例和紧急比例可以通过环境变量调整。

此外，项目使用环境变量控制多种行为（如是否允许 bash、最大并行工具数量、紧急压缩阈值、是否显示模型路由等），使系统在不同部署环境下能够自由调优。

## 总结与启示

`agent‑loop` 项目以一个简单的流式代理循环为起点，逐步构建了一个功能丰富、可扩展且安全的 AI 代理平台。从核心的模型调用与 SSE 输出，到完善的工具管理、记忆压缩、子代理调度、技能系统、任务工作流，再到工作树隔离和团队协作，每个模块都围绕着如何提升代理的自主行动力和可控性而设计。

- **架构清晰**：以“调用→检查→执行→循环”的代理控制循环为主轴，所有功能都围绕这一流程展开；
- **分层设计**：通过工具系统、子代理和技能加载，将复杂的操作解耦，并依据风险和作用划分权限；
- **可扩展记忆**：多层压缩策略让长对话可持续进行，同时提供溢出文件和摘要，方便模型回顾；
- **灵活调度**：并行执行只读工具、后台任务和工作树隔离，提高效率并支持团队协作；
- **高度可配置**：环境变量驱动的配置使得代理可以适应不同 API 提供商和安全策略。

这些设计经验不仅适用于本项目，也为构建其他基于 LLM 的代理系统提供了可借鉴的思路：从简单的 MVP 出发，不断增添模块以满足新需求，同时保持良好的架构和安全边界。在未来，我们可以继续探索更精细的权限管理、智能计划与优先级调度、更强大的团队协作机制以及插件化的技能生态等方向，让代理系统真正成为开发者和团队的高效助理。
